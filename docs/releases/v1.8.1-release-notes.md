# v1.8.1 release notes

## Summary

Patch release fixing a **crwlr regression in v1.8.0**: `crwlr` could not connect
through the new auth gateway because crawl4ai runs its own unauthenticated
readiness probe. ask-search and cdp-download were unaffected.

## The bug

crawl4ai's `_verify_cdp_ready()` issues its own `aiohttp` GET to
`<cdp_url>/json/version` *before* connecting — a code path entirely separate
from `connect_over_cdp`. The v1.8.0 crwlr patch only injected `X-API-Key` on the
`connect_over_cdp` WebSocket handshake, so this probe hit the gateway with no
key, got a 401, and crawl4ai aborted with `CDP verification failed after 5
attempts` — never reaching the (correctly patched) WS connect.

This is exactly the unverified item the v1.8.0 final review flagged: the CDP
HTTP-discovery leg was never exercised end-to-end before release.

## The fix

`skills/crwlr/scripts/crwlr` now also patches `aiohttp.ClientSession._request`
to inject `X-API-Key`, **scoped to the CDP host only** (the key is never sent to
third-party sites crwlr crawls). Combined with the existing `connect_over_cdp`
patch, both legs of the CDP path now carry the key.

## Regression test

`backend/tests/test_integration.sh` gains a crwlr end-to-end check: it runs the
real script against the authed gateway and fails if it sees
`CDP verification failed` or doesn't return page content. Verified passing
against the live NAS backend.

## Verification

- Integration (live NAS): all three ports enforce auth + crwlr fetches
  example.com through the gateway. PASS.
- Offline `skills/tests/test_auth_injection.py`: all three scripts PASS.
